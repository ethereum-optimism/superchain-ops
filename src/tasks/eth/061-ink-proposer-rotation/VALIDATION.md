# Validation

This document can be used to validate the inputs and result of the execution of the transaction which you are signing.

The steps are:

1. [Validate the Domain and Message Hashes](#expected-domain-and-message-hashes)
2. [Transaction Inputs](config.toml): inputs can be verified in the config.toml file.
3. State Changes: the template's `_validate` block asserts the new `gameArgs(1)` and unchanged `gameImpls(1)`. State changes can also be reviewed in Tenderly via the link printed during simulation.

## Understanding Task Calldata

The task calls `setImplementation` **once** on the Ink mainnet DisputeGameFactory (`0x10d7B35078d3baabB96Dd45a9143B94be65b12CD`, expected v1.6.1), for PDG (game type 1). The implementation address does not change (live `gameImpls(1)`); only the `gameArgs` blob changes, and within it **only the proposer**.

The 164-byte `gameArgs(1)` blob has layout `prestate(32) | vm(20) | ASR(20) | delayedWETH(20) | chainId(32) | proposer(20) | challenger(20)`. It is computed from the **live** on-chain `gameArgs(1)` with the proposer swapped to the OPE proposer, so it must be read at sign time:

```bash
# 1. Read the live blob and the live impl:
cast call 0x10d7B35078d3baabB96Dd45a9143B94be65b12CD "gameArgs(uint32)(bytes)" 1 --rpc-url <MAINNET_RPC>
cast call 0x10d7B35078d3baabB96Dd45a9143B94be65b12CD "gameImpls(uint32)(address)" 1 --rpc-url <MAINNET_RPC>
# 2. Rebuild the calldata with bytes 124..144 (proposer) replaced by the OPE proposer:
cast calldata "setImplementation(uint32,address,bytes)" 1 <LIVE_IMPL> <NEW_BLOB>
# selector: 0xb1070957
```

> [!IMPORTANT]
> `SetDisputeGameArgs` reads `gameArgs(1)` live and overrides only the proposer, so the blob is recomputed at sign time — re-run `just simulate` and re-verify the proposer bytes (124–144) equal the OPE proposer and the challenger bytes (144–164) equal `9ba6e03d8b90de867373db8cf1a58d2f7f006b3a` (unchanged). This task touches game type 1 (dormant permissioned fallback), not the active game type 8.

The outer Multicall3 blob is assembled by the framework and printed during simulation; confirm it contains only the single `setImplementation` call above.

## Task State Changes

### `0x10d7B35078d3baabB96Dd45a9143B94be65b12CD` (DisputeGameFactoryProxy) — Chain ID 57073

- `gameArgs(1)` (PERMISSIONED_CANNON) proposer updated from the current Gelato proposer (CONFIRM on-chain) → OPE proposer. Challenger (`0x9ba6e03d…006b3a`), prestate, vm, ASR, delayedWETH, chainId, and the implementation address are unchanged by this task.
- Game types 0 (CANNON) and 8 (CANNON_KONA) are untouched.

The proposer occupies the chainId/proposer boundary, so the write lands across two packed `gameArgs` storage slots (challenger bytes are preserved in both). The exact storage keys and before/after values are printed by `just simulate` — capture them here once generated.

### Signer safes

`Security Council` and `Foundation Upgrade Safe` nonces increment by 1 (nested execution through the L1 ProxyAdminOwner `0x5a0Aae59D09fccBdDb6C6CcEB07B7279367C3d2A`).
